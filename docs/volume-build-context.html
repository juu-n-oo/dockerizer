AIPub Volume ↔ Build Context 경로 모델

같은 PVC를 "파일 브라우저"와 "Kaniko 빌드"가 서로 다른 방식으로 마운트한다 — 왜 경로가 어긋나는지, 어떻게 맞추는지

1 큰 그림 — 하나의 PVC, 두 명의 소비자

AIPubVolume CR 은 내부적으로 하나의 PVC(예: pjw-volume-xxxx)로 resolve 된다. 이 PVC 는 "데이터가 실제로 저장된 디스크"다. 우리가 다루는 두 화면은 같은 PVC를 보지만 서로 다른 pod 에서, 서로 다른 위치에 마운트해서 본다.

┌─────────────────────────────┐ │ PVC (실제 저장 데이터) │ ← 진실의 원본. 경로는 "PVC 루트 기준" └─────────────────────────────┘ ▲ ▲ mount │ (읽기) mount │ (읽기전용) │ │ 파일 브라우저 Kaniko 빌드 pod 접근 pod 안 /data 에 마운트 /workspace 에 마운트 (Dockerfile 작성 시 파일 선택) (--context=dir:///workspace)

두 소비자가 PVC 의 어느 부분을(subPath) 어디에(mountPath) 붙이느냐가 다르면, 같은 파일이라도 "보이는 경로"가 달라진다. 이게 혼란의 근원이다.

2 마운트의 기본 규칙 (mountPath / subPath)

k8s 에서 PVC 를 pod 에 붙일 때 두 가지를 정한다:

  • mountPath — pod 안에서 어느 경로에 보이게 할지
  • subPath (선택) — PVC 전체가 아니라 그 하위 디렉터리만 붙일지
PVC 안의 경로 <subPath>/X ──마운트──▶ pod 안의 경로 <mountPath>/X

즉 마운트하면 subPath 부분은 "잘려 나가고", mountPath 가 새 루트가 된다. 같은 파일이 pod 마다 다른 절대경로로 보이는 이유다.

예) PVC 안 /team-a/data/x.txt 를 subPath=team-a/data, mountPath=/data 로 붙이면 → pod 안에서는 /data/x.txt 로 보인다. team-a/data 라는 앞부분은 사라진다.
3 같은 파일, 세 가지 시점

AIPub 볼륨에 all_conversations.txt 와 temp/common.zip 이 있고, 파일 브라우저에서는 /data/... 아래로 보이는 상황을 예로 든다. (PVC 내부의 실제 subPath 는 S 로 표기 — 4·6장에서 실제 값 확인 방법 제공)

① PVC 실제 저장
진실의 원본 (PVC 루트 기준)
PVC:/ └─ S/ ← AIPub 가 쓰는 내부 subPath ├─ all_conversations.txt └─ temp/ └─ common.zip

PVC 루트 기준 실제 경로:
S/all_conversations.txt
S/temp/common.zip

② 파일 브라우저 시점
접근 pod: subPath=S → /data
/ (접근 pod 파일시스템 루트) └─ data/ ← S 가 여기 마운트됨 ├─ all_conversations.txt └─ temp/ └─ common.zip

사용자가 보는 경로:
/data/all_conversations.txt
/data/temp/common.zip
→ 여기서 data 는 PVC 안 폴더가 아니라 마운트 지점

③ Kaniko 빌드 시점
빌드: subPath 없이 → /workspace
/workspace/ (= PVC 루트 그대로) └─ S/ ├─ all_conversations.txt └─ temp/ └─ common.zip

Kaniko 가 보는 실제 경로:
/workspace/S/all_conversations.txt
→ /data 가 아니라 S 가 앞에 붙는다

핵심: 브라우저의 /data 와 빌드의 /workspace 는 같은 PVC 의 같은 지점이 아니다. 브라우저는 subPath=S 를 잘라낸 뒤 /data 를 루트로 보여주고, 빌드는 PVC 루트를 통째로 /workspace 에 붙인다. 그래서 브라우저에서 본 /data/... 를 그대로 COPY 에 쓰면 어긋난다.
4 COPY 가 경로를 해석하는 규칙

Dockerfile 의 COPY <src> <dst> 에서 <src> 는 빌드 컨텍스트 루트(/workspace) 기준 상대경로다. Kaniko 는 다음 위치에서 파일을 찾는다:

찾는 위치 = /workspace/<src> = PVC:/ <buildContextSubPath> / <src>

따라서 빌드가 성공하려면 이 한 줄이 성립해야 한다:

<buildContextSubPath> / <src> == 파일의 PVC-루트-기준 실제 경로

즉 "빌드 시 마운트하는 subPath" 와 "COPY 에 적는 경로" 를 합치면, PVC 안 실제 위치와 정확히 같아야 한다. 둘 중 하나만 알아도 나머지를 정할 수 있다.

5 그래서 지금 왜 실패했나

두 번의 시도가 모두 실패했다 (둘 다 buildContextSubPath 미설정 = PVC 루트를 /workspace 에 마운트):

시도한 COPYKaniko 가 찾은 위치결과
COPY data/all_conversations.txt/workspace/data/all_conversations.txt = PVC:/data/...없음 (PVC 루트에 data/ 폴더 없음)
COPY all_conversations.txt/workspace/all_conversations.txt = PVC:/all_conversations.txt없음 (PVC 루트에 파일 없음)
둘 다 실패했다는 건 → 파일이 PVC 루트의 data/ 에도, PVC 루트 바로 아래에도 없다는 뜻. 실제 데이터는 PVC 내부의 다른 subPath(S) 아래에 있고, 빌드는 그 S 를 건너뛴 채 PVC 루트만 마운트했기 때문이다. 브라우저가 보여준 /data 는 그 S 를 가린 "접근 pod 의 마운트 지점"일 뿐이다.
6 올바르게 맞추는 법
STEP 1. 실제 PVC-기준 경로(=S) 를 알아낸다

두 가지 방법 중 하나로 "파일이 PVC 루트 기준 어디에 있는지" 를 확인한다.

방법 A — 접근 pod 의 마운트 설정 확인 (브라우저가 쓰는 그 pod):

kubectl get pod <pvcName> -n <project-ns> \ -o jsonpath='{range .spec.containers[*].volumeMounts[*]}{.name}{" mountPath="}{.mountPath}{" subPath="}{.subPath}{"\n"}{end}' # mountPath=/data 인 줄의 subPath 값이 곧 S 다. # 예: subPath=team-a/data → S = team-a/data

방법 B — 빈 빌드로 직접 들여다보기 (COPY 없이, 같은 볼륨을 컨텍스트로):

FROM aipub-harbor.cluster10.idc1.ten1010.io/common/ubuntu:latest RUN ls -laR /workspace || true CMD ["sleep", "infinity"]

빌드 로그에 /workspace 의 실제 트리가 찍힌다 → 파일이 PVC 루트 기준 어디 있는지 눈으로 확인. (COPY 가 없으니 실패 없이 구조만 보여준다.)

STEP 2. subPath 와 COPY 경로를 맞춘다

S 를 알았다면, 두 가지 동등한 방법 중 하나를 쓴다.

방법 ① subPath 로 맞추기 (권장)

빌드 다이얼로그의 서브 경로 에 S 를 넣는다. 그러면 /workspace 가 곧 브라우저의 /data 와 같아져서, COPY 는 /data 이후 경로만 적으면 된다.

# buildContextSubPath = S COPY all_conversations.txt /workspace/ COPY temp/common.zip /workspace/temp/

→ 찾는 위치 PVC:/S/all_conversations.txt ✔

방법 ② subPath 없이, COPY 에 전부 적기

서브 경로를 비우고(=PVC 루트 마운트), COPY 에 S 까지 포함한 전체 경로를 적는다.

# buildContextSubPath = (비움) COPY S/all_conversations.txt /workspace/ COPY S/temp/common.zip /workspace/temp/

→ 찾는 위치 PVC:/S/all_conversations.txt ✔ (동일)

핵심 한 줄: "빌드 subPath + COPY 경로" 를 합치면 항상 "파일의 PVC 루트 기준 실제 경로(=S/…)" 와 같게 만든다. 어디서 끊느냐(subPath 로 흡수 vs COPY 에 명시)는 취향 차이일 뿐 결과는 같다.
7 참고 — 목적지(<dst>) 는 별개다

COPY <src> <dst> 에서 <dst> 는 빌드되는 이미지 안의 경로다. 컨텍스트(/workspace)와 무관하며, 실패 원인과도 무관하다. 예:

COPY all_conversations.txt /workspace/ # 이미지 안 /workspace/all_conversations.txt 로 복사 COPY temp/common.zip /app/data/ # 이미지 안 /app/data/common.zip 로 복사

즉 왼쪽(src)=컨텍스트(볼륨)에서 읽을 위치, 오른쪽(dst)=이미지에 쓸 위치. 이번 오류는 전부 왼쪽(src)·마운트 문제이고 오른쪽(dst)은 아무 상관 없다.

✓ 한 장 요약
  • 브라우저 경로 ≠ 빌드 경로. 브라우저는 subPath=S 를 잘라 /data 로 보여주고, 빌드는 PVC 루트를 /workspace 에 붙인다.
  • COPY src 는 /workspace(컨텍스트 루트) 기준 상대경로. 찾는 위치 = PVC:/<subPath>/<src>.
  • 성립 조건: <subPath> + <src> == 파일의 PVC-루트-기준 실제 경로(S/…).
  • 지금 실패 이유: 실제 데이터가 PVC 의 S 아래 있는데 빌드는 S 를 건너뛴 PVC 루트만 마운트 → data/… 도 … 도 모두 못 찾음.
  • 해결: ① 접근 pod 의 subPath(=S) 를 확인 → ② 빌드 서브경로에 S 를 넣고 COPY 는 그 이후만 적기 (또는 subPath 비우고 COPY 에 S/… 전체 적기).
추가 메모: 현재 UI 는 파일 브라우저가 보여준 /data/... 를 그대로 COPY 로 만들면서 접근 pod 의 마운트 지점(data)을 볼륨 내부 경로로 오인한다. 또 빌드 서브 경로 입력은 파일 선택과 자동 연동되지 않는다. 근본 수정은 "브라우저가 보여주는 기준점(= 접근 pod 의 mountPath/subPath)" 정보를 빌드 마운트에 자동 반영하는 것이다. (별도 작업)
